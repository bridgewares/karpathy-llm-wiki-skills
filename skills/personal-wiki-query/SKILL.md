# Personal Wiki Query

这个 skill 用于基于个人 wiki 中**已经存在的知识层与来源层**回答问题、做比较、做总结，并在必要时补回原始资料证据。

目标不是做 init、ingest 或 lint，而是尽量复用现有 wiki 沉淀，给出清晰、结构化、可继续积累的回答。

## 适用场景

当用户表达以下意图时使用：

- “这个 wiki 里现在怎么回答这个问题？”
- “帮我看看这个主题当前有哪些结论和证据”
- “比较这几页 / 这几份来源的观点”
- “总结一下这个问题目前在知识库里的答案”
- “先基于现有 wiki 回答，不够再回原文看”
- “把这个问题当前的答案整理一下，并视情况写回 wiki”

如果用户要做的是：

- 初始化骨架 → 这不是本 skill
- 导入具体来源、移动文件、登记 `raw-index.md`、建 source page → 这不是本 skill
- 检查孤儿页、交叉引用、一致性、`needs-update` → 这不是本 skill
- 只编辑单个 topic / concept / person / question 页面而没有 query 意图 → 这不是本 skill

## 工作模式

本 skill 默认支持三类 query：

1. **直接问答**
   - 用户围绕现有 wiki 内容提问。
   - 目标是给出当前结论、依据页面和证据，而不是重做 ingest。

2. **对比 / 综合**
   - 用户要求比较多个页面、多个来源，或归纳某个主题当前的共识、分歧与空白。
   - 目标是形成结构化比较或阶段性总结。

3. **回写增强**
   - 用户明确要求把 query 结果写回 `wiki/questions/` 或 `wiki/syntheses/`。
   - 或者 query 结果明显具有长期复用价值，应主动建议沉淀。
   - 如果是否落盘仍有判断空间，则先询问用户，而不是静默写入。

## 核心原则

1. **先看 `index.md`，再读 `wiki/`**
   - 先通过导航判断相关区域。
   - 不要一上来就直接扎进 `raw/`。

2. **优先复用已有知识页**
   - 优先读取相关 `wiki/questions/`、`wiki/syntheses/`、`wiki/topics/`、`wiki/concepts/`、`wiki/people/`、`wiki/projects/`、`wiki/timelines/`、`wiki/sources/`。
   - query 的目标是复用和整合，而不是每次都重新整理来源。

3. **只有在 wiki 信息不足时才回到 `raw/`**
   - 当 wiki 证据不够、存在冲突待核对、或用户明确要求看原文时，再去 `raw/library/` 补证据。
   - `raw/` 是补证据层，不是默认起点。

4. **区分结论、证据、冲突与未解决问题**
   - 回答中要尽量明确：当前结论是什么、依据是什么、哪里有冲突、还缺什么。
   - 这样后续才容易判断是否值得沉淀回 wiki。

5. **默认回答，不默认落盘**
   - 默认先完成回答。
   - 不因为一次回答看起来不错就自动写回 wiki。

6. **高价值结果主动建议沉淀**
   - 对稳定问题、跨来源比较、长期复用型总结，应主动建议写回。
   - 如果用户原 prompt 已明确要求写回，则直接执行，不再额外确认。
   - 如果价值明显但是否落盘仍存在判断空间，则先问用户。

7. **不把猜测写成事实**
   - 缺证据时要说明不足。
   - 发现来源或页面之间有冲突，要明确标出冲突，不强行统一成单一说法。

## query 目标结果

一次成功 query 之后，通常应当得到这些结果：

- 用户问题得到了清晰回答
- 回答优先基于现有 `wiki/` 页面，而不是跳过知识层直接读 raw
- 如有必要，回答补充了来自 `raw/` 的证据
- 回答中明确区分了结论、依据、冲突和空白
- 如果结果值得长期保留，给出了沉淀建议
- 如果用户明确要求落盘，则更新了对应 `wiki/questions/...` 或 `wiki/syntheses/...` 页面
- 如执行了写回，`log.md` 已追加记录

## 执行流程

按下面顺序执行。

### 1. 判断是否属于 query

先判断用户要做的是不是以下事情：

- 询问某个问题当前在 wiki 中的答案
- 检索与某个主题有关的现有页面
- 比较多个来源页、主题页、概念页、人物页或综合页
- 归纳一个主题当前的主要结论、证据和冲突
- 核对 wiki 说法是否能被已有来源支持

如果本质上是 init、ingest、lint 或普通页面编辑，就不要按本 skill 处理。

### 2. 检查 `index.md`

先查看：

- 当前重点主题入口
- 相关知识页区域大致在哪
- 是否已有明显对应的问题页、综合页或来源页路径

### 3. 优先读取相关 `wiki/` 页面

优先从这些地方找答案：

- `wiki/questions/...`
- `wiki/syntheses/...`
- `wiki/topics/...`
- `wiki/concepts/...`
- `wiki/people/...`
- `wiki/projects/...`
- `wiki/timelines/...`
- `wiki/sources/...`

读取时重点关注：

- 当前结论
- 已引用的来源
- 存在的冲突信息
- 页面中标出的未解决问题

### 4. 必要时回到 `raw/` 补证据

如果出现以下情况，再去 `raw/library/`：

- wiki 信息明显不足
- wiki 有结论但证据链不够清楚
- 不同页面之间存在冲突，需要核对原始来源
- 用户明确要求回到原文或原始资料核对

补证据时应尽量围绕当前问题，不把 query 变成一次新 ingest。

### 5. 生成结构化回答

默认回答应尽量包含：

- `结论`
- `依据页面 / 来源`
- `冲突或不确定`
- `下一步`

如果只是轻量查询，也至少要给出：

- 当前答案
- 主要依据来自哪些页面或来源

### 6. 判断是否建议沉淀

当回答具有以下特征时，主动建议写回 wiki：

- 一个会被反复问到的稳定问题
- 一次跨多个来源或页面的高价值比较
- 对某个主题形成了较完整的阶段性总结
- 回答明确补齐了过去页面中的重要缺口

写回落点规则：

- `wiki/questions/...`：更适合“某个问题当前怎么回答”
- `wiki/syntheses/...`：更适合“跨来源综合、比较、专题总结”

如果用户已经明确要求“写回 / 记下来 / 落盘到 wiki”，则直接执行对应写回。
如果是否落盘仍有判断空间，则先问用户，而不是直接写入。

### 7. 执行写回与记日志（仅在需要时）

如果用户明确要求落盘，或已获得确认，应：

- 优先更新已有 `wiki/questions/...` 或 `wiki/syntheses/...` 页面
- 只有确有必要时才新建最小新页
- 保持结构化内容，而不是把聊天记录原样贴进去
- 在 `log.md` 追加 query / synthesis 类记录

### 8. 最小验证

完成后至少检查：

- 是否优先使用了 `wiki/` 而不是直接跳到 `raw/`
- 如果读取了 `raw/`，是否确实是为当前 query 补证据
- 回答是否区分了结论、依据、冲突或空白
- 如果建议了沉淀，建议的落点是否合理
- 如果执行了写回，目标页面和 `log.md` 是否已更新

## 输出风格

默认用简洁中文汇报，避免长篇解释。

推荐回答结构：

```md
结论：
- ...

依据页面 / 来源：
- ...

冲突或不确定：
- ...

下一步：
- ...

沉淀建议：
- 可考虑写回 `wiki/questions/...` / `wiki/syntheses/...`
```

如果已经执行了落盘，推荐补充：

```md
已更新：
- wiki/questions/...
- wiki/syntheses/...
- log.md
```

## 不要这样做

- 不要把 query 做成 ingest：不要移动文件、不要整理 `raw/inbox/`、不要维护来源入库状态
- 不要在 `wiki/` 已足够时直接跳到 `raw/`
- 不要把轻量一次性回答默认写入 wiki
- 不要在用户未明确要求且价值判断仍模糊时静默落盘
- 不要把冲突信息强行抹平成单一结论
- 不要把猜测写成来源或页面已经明确支持的事实
- 不要把普通单页编辑任务混进 query

## 示例触发

**示例 1**
用户：`这个 wiki 里现在怎么回答“如何减少知识漂移”？先基于已有页面说。`

动作：使用本 skill，先看 `index.md` 和相关 `wiki/questions/...`、`wiki/topics/...` 页面，给出结构化回答。

**示例 2**
用户：`比较一下 Agent Memory 这个主题下几份来源的共识和分歧。`

动作：使用本 skill，基于多个 `wiki/sources/...`、`wiki/topics/...` 或 `wiki/syntheses/...` 页面做比较和归纳。

**示例 3**
用户：`先根据现有 wiki 回答，如果证据不够，再回原文补。`

动作：使用本 skill，优先读取 `wiki/`，只有在信息不足时再去 `raw/library/` 补证据。

**示例 4**
用户：`把这个问题当前答案整理出来，并写回 wiki/questions。`

动作：使用本 skill，先回答，再直接更新对应 `wiki/questions/...` 页面并记日志，不再额外确认是否落盘。
